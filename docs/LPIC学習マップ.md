# LPIC 学習マップ（CLI_Noir ゲーム対応表）

このファイルは、LPIC Level 1（101/102）の出題範囲と CLI_Noir ゲーム内の対応を整理したもの。  
Mission やストーリー設計時に「どのコマンド・概念をどう体験させるか」を決める基準として使う。

---

## 読み方

| 列 | 意味 |
|---|---|
| LPIC 項目 | LPIC 出題範囲の分類 |
| コマンド / 概念 | 実際に学ぶ対象 |
| 実 PC での意味 | 本来の Linux での役割 |
| ゲーム内の対応 | CLI_Noir でどう体験させるか |
| Phase | MVP（Phase1）か将来拡張か |

## 1. ファイルとディレクトリの操作（LPIC 103.3）

ゲームの基本動作。Mission1 で集中的に学ぶ。

| コマンド | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| `ls` | ディレクトリ内容の一覧表示 | 部屋や場所にある物（ファイル/フォルダ）を見る | 1 |
| `ls -l` | 詳細表示（権限・サイズ・日時） | ファイルの詳細情報を確認する | 1 |
| `ls -a` | 隠しファイルを含めて表示 | 隠された手がかりファイルを発見する | 1 |
| `cd` | ディレクトリ移動 | 場所を移動する（事務所→机→公園） | 1 |
| `cd ..` | 親ディレクトリへ移動 | 1つ前の場所に戻る | 1 |
| `cd /` | ルートへ移動 | 事務所（ホーム）に一気に戻る | 1 |
| `pwd` | 現在のディレクトリを表示 | 今いる場所を確認する | 1 |
| `mkdir` | ディレクトリ作成 | メモ用の部屋（フォルダ）を作る | 1 |
| `touch` | 空ファイル作成 | メモファイルを新規作成する | 1 |
| `cp` | ファイルコピー | 証拠のバックアップを取る | 2 |
| `mv` | ファイル移動/リネーム | 証拠を別の場所に移す / 名前を変える | 2 |

**ゲーム設計のポイント:**  
`ls` で「何があるか見る」→ `cd` で「移動する」→ `cat` で「中身を読む」が、実 PC でも CLI_Noir でも同じ行動パターンになる。プレイヤーは意識せず実務と同じ操作手順を身につける。

## 2. ファイルの内容表示と編集（LPIC 103.2 / 103.4）

Mission1〜2 で使う。ファイルの「読み書き」を学ぶ。

| コマンド | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| `cat` | ファイル内容を全表示 | 手がかりファイルを読む | 1 |
| `less` | ページ送りで閲覧 | 長い証拠文書をページ送りで読む | 1 |
| `echo` | 文字列を出力 | メモを書く / ファイルに追記する | 1 |
| `echo "x" > file` | ファイルに上書き | 報告書に内容を書き込む | 1 |
| `echo "x" >> file` | ファイルに追記 | 報告書に情報を追加する | 1 |

**ゲーム設計のポイント:**  
`echo "名前" > businesscard.txt` は、実 PC では「ファイルにテキストを書き込む」操作。ゲーム内では「名刺に自分の名前を書く」行為。操作と結果の意味が一致している。

## 3. テキスト検索とフィルタリング（LPIC 103.7）

Mission2〜3 の中核。LPIC で最も躓きやすい領域。

| コマンド | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| `grep` | パターンに一致する行を検索 | ファイルから特定の手がかりを探す | 1 |
| `grep -i` | 大文字小文字を区別しない検索 | 曖昧な情報でも見つけられる | 1 |
| `grep -E` | 拡張正規表現で検索 | 複雑なパターンの手がかりを抽出する | 1 |
| `grep -r` | ディレクトリ再帰検索 | 場所全体から一括で手がかりを探す | 2 |
| `find` | 条件に合うファイルを検索 | 広いエリアからファイルを探し出す | 1 |
| `find -name` | 名前でファイル検索 | 「catinfo.txt はどこにある？」 | 1 |
| `find -type` | 種類（ファイル/ディレクトリ）で検索 | 「ディレクトリだけ一覧にしたい」 | 2 |
| `sort` | 行をソート | 集めた情報を整理する | 1 |
| `uniq` | 重複行を除去 | 重複した手がかりをまとめる | 1 |
| `awk` | パターンマッチ + テキスト加工 | 構造化データから必要な列だけ抜き出す | 1 |

**ゲーム設計のポイント:**  
`grep "Height" catinfo.txt` は、実 PC では「ファイルから Height を含む行を抽出」。ゲーム内では「猫の情報ファイルから身長データを見つける」。検索対象が「ログファイル」から「証拠ファイル」に変わっただけで、操作と思考プロセスは同一。

## 4. ファイルシステムとパス（LPIC 104.1 / 104.2）

全 Mission を通じて自然に学ぶ概念。

| 概念 | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| 絶対パス | `/` から始まる完全なパス | `/root/park/swing/catinfo.txt` のように場所を正確に指定する | 1 |
| 相対パス | 現在地からの相対指定 | `cd desk` で「今いる場所から机へ」 | 1 |
| ルートディレクトリ `/` | ファイルシステムの最上位 | 探偵事務所のビル全体 | 1 |
| ホームディレクトリ | ユーザーの作業起点 | 探偵事務所（`/root`） | 1 |
| ディレクトリ階層 | ツリー構造 | 事務所 > 机 / ドア / 公園 > 遊具 | 1 |
| 隠しファイル `.xxx` | `.` で始まるファイル | 隠された証拠や秘密メモ | 2 |

**ゲーム設計のポイント:**  
Mission2 の完了条件に「猫ファイルの絶対パスを記述する」を入れることで、絶対パスの概念を「使わないとクリアできない」形で自然に定着させる。

## 5. リモート接続（LPIC 110.1）

Mission3 の中核。サーバー運用の入り口。

| コマンド / 概念 | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| `ssh` | リモートサーバーに安全に接続 | 遠くの場所（遊園地）に出張する | 1 |
| `exit` | リモートセッションを終了 | 出張先から事務所に帰る | 1 |
| ローカル / リモート | 手元 PC とサーバーの区別 | 事務所（local）と出張先（remote） | 1 |
| ホスト名 | 接続先サーバーの識別名 | `amusement_park` = 遊園地 | 1 |
| 接続拒否 | 認証失敗 / ホスト不明 | `Permission denied` / `Host not found` | 1 |

**ゲーム設計のポイント:**  
`ssh amusement_park` で画面がフェード遷移し、背景が遊園地に変わる。「local と remote は別の世界」という感覚を視覚的に体験させる。`cd` では戻れず `exit` でしか帰れないルールが、実際の SSH セッション管理と同じ。

## 6. バージョン管理の基礎概念（LPIC 対象外だが実務必須）

疑似 Git として全 Mission で使用。

| コマンド | 実 PC での意味 | ゲーム内の対応 | Phase |
|---|---|---|---|
| `git status` | 作業状態の確認 | 報告書の進捗を確認する | 1 |
| `git add` | 変更をステージング | 証拠を報告書に添付する | 1 |
| `git commit -m` | 変更を記録 | 報告書をセーブする（何度でも可） | 1 |
| `git push` | リモートに送信 | 報告書を本部に提出する（クリア判定） | 1 |

| 概念 | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| ステージング | commit 前の準備領域 | 報告書に添付する証拠を選ぶ段階 |
| コミット履歴 | 変更の記録 | セーブデータの一覧 |
| push | リモートリポジトリへ反映 | 提出して次のステージへ |

**ゲーム設計のポイント:**  
`add → commit → push` の順序制約を体で覚えさせる。commit が「セーブ」、push が「提出」という対応により、Git ワークフローの「なぜこの順序なのか」が直感的に理解できる。

## 9. テキスト処理の拡張（LPIC 103.2 / 103.4）— Level 5「情報屋」/ Phase 2

§ 8 のパイプと組み合わせ、「1つのコマンドで終わらせず、流れ作業で情報を捌く」段階。

| コマンド | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| `head` | ファイルの先頭N行を表示 | 分厚い調書の冒頭だけ確認する（全部読む必要はない、と学ぶ） |
| `tail` | ファイルの末尾N行を表示 | 日誌・ログの「最新の記録」を確認する |
| `tail -f` | 追記をリアルタイム監視 | 盗聴ログを張り込み監視する（ライブ表示演出） |
| `wc` | 行数・単語数・バイト数を数える | 証拠の分量を数える。「容疑者リストは全部で何人？」 |
| `cut` | 行から特定の列を切り出す | 名簿から「電話番号の列だけ」抜き出す |
| `paste` | 複数ファイルの行を横に結合 | 破られたメモを貼り合わせて復元する |
| `tr` | 文字の置換・削除 | 換字式暗号（シーザー暗号など）を解読する |
| `sed` | ストリーム編集（置換・削除・抽出） | 改ざんされた文書を元の記述に修正する |
| `diff` | 2ファイルの差分を表示 | 原本と偽造文書を見比べ、改ざん箇所を特定する |
| `nl` | 行番号を付けて表示 | 証拠文書に整理番号を振る |
| `tee` | 出力を画面とファイルの両方へ書く | 確認しながら同時に記録も残す |
| `xargs` | 標準入力を次のコマンドの引数に変換 | 見つけた証拠リストをまとめて次の処理にかける |

**ゲーム設計のポイント:**
「数万行のログ」を配置し、`cat` で全部読むのは事実上不可能な状況を作る。
`grep | sort | uniq -c` でしか現実的に解けない Mission により、パイプの**必要性**を体感させる。

## 10. プロセス管理（LPIC 103.5）— Level 6「監視者」/ Phase 2

「PC の中では常に何かが動いている」ことを体験する。evaluator に仮想プロセステーブルを持たせる。

| コマンド | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| `ps` | 実行中プロセスの一覧表示 | 「今この建物の中で動いている者」を一覧する。PID＝容疑者番号 |
| `ps aux` | 全ユーザーの全プロセスを詳細表示 | 建物全体の人物・挙動の詳細リスト |
| `top` | プロセスをリアルタイム監視 | 監視カメラモニター（CPU 使用率＝怪しい動きの激しさ） |
| `kill <PID>` | プロセスに終了シグナルを送る | 盗聴プログラムを停止させる（＝確保）。※仮想プロセス限定 |
| `pgrep` / `pkill` | 名前でプロセスを検索/終了 | 名前から容疑プロセスを割り出して止める |
| `jobs` / `fg` / `bg` | ジョブの一覧/前面/背面切替 | 並行して進めている捜査タスクの切り替え |
| `free` / `uptime` | メモリ使用量 / 稼働時間の確認 | 建物の稼働状況・「いつから動いているか」を調べる |

**ゲーム設計のポイント:**
`kill` は実PCでも「プロセスを止める」以上のことはしない、と正しく学ばせる。
正規プロセスを kill すると警告＋巻き戻し（即失敗にしない方針を維持）。

## 11. ユーザーと権限の拡張（LPIC 104.5 / 107.1 / 110.2）— Level 7「潜入捜査官」/ Phase 2

§ 7 の `chmod` / `chown` を本格導入し、ユーザー概念まで広げる。

| コマンド / 概念 | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| `chmod +x` | 実行権限の付与 | `case_file.sh` が「実行できない」状態から自力で解除する |
| `umask` | 新規ファイルのデフォルト権限 | 「作るメモに最初からかかる鍵」の設定 |
| `whoami` / `id` | 現在のユーザー名 / UID・グループ確認 | 変装中の「今の自分は誰か」を確認する |
| `su <user>` | 別ユーザーに切り替え | 潜入捜査のための変装。パスワード＝合言葉 |
| `who` / `w` | ログイン中ユーザーの確認 | 「今この建物に誰が入っているか」を調べる |
| `useradd` / `passwd` | ユーザー作成 / パスワード設定 | 協力者の身分証を発行する（Phase3 候補） |
| `sudo` | 管理者権限で実行 | **原則封印**（denylist 維持）。最終 Mission のイベント演出限定で検討 |

**ゲーム設計のポイント:**
`ls -l` の `rwxr-xr--` 表示を「鍵の刻印」として読む訓練を Mission に組み込む。
数値モード（`chmod 755`）は記号モード習得後の加点要素にする。

## 12. アーカイブ・リンク・鑑識（LPIC 103.3 / 104.6）— Level 8「証拠管理官」/ Phase 2

## 13. ネットワーク（LPIC 109.x / 110.1）— Level 9「追跡者」/ Phase 2

Mission3 の `ssh` 体験を発展させ、「接続する前に調べる」を学ぶ。

| コマンド | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| `ping` | 相手ホストへの疎通確認 | 「その場所は今も存在するか」生存確認の電話を鳴らす |
| `ip addr` | 自分の IP アドレス確認 | 自分の現在地（住所）を確認する |
| `ss` | 開いているポート・接続の確認 | 「建物のどの扉が開いているか」を調べる |
| `dig` / `host` | ドメイン名から IP を引く（DNS） | 通称（ホスト名）から実際の住所（IP）を割り出す |
| `hostname` | 自ホスト名の確認 | 今いる拠点の名前を確認する |
| `traceroute` | 相手までの経路を表示 | 犯人サーバーまでの追跡ルートを地図に描く |
| `curl` | HTTP リクエスト送信（mock 限定） | 本部データベース API への照会 |

**ゲーム設計のポイント:**
「`dig` で IP を突き止める → `ping` で生存確認 → `ssh` で突入」という実務と同じ順序を
Mission の必須フローにする。DNS の概念を追跡劇として体験させる。

## 15. シェルスクリプト基礎（LPIC 105.1 / 105.2）— Level 11「参謀」/ Phase 2

「手作業の捜査を手順書（スクリプト）に落とす」最終段階。
これまで実行するだけだった `case_file.sh` の中身を、自分で書く側に回る。

| コマンド / 構文 | 実 PC での意味 | ゲーム内の対応 |
|---|---|---|
| `sh <file>` | シェルスクリプトの実行 | 手順書どおりに捜査を自動実行する |
| 変数 `NAME=値` / `$NAME` | 値に名前を付けて保存・参照 | 捜査メモに名前を付けて持ち歩く |
| `$?` | 直前コマンドの終了コード（0=成功） | 「さっきの捜査は成功したか？」への部下の返事 |
| `if` / `then` / `fi` | 条件分岐 | 「証拠が見つかったら報告する」を手順書に書く |
| `for` / `do` / `done` | 繰り返し処理 | 全ファイルへの聞き込みを自動化する |
| `test` / `[ ]` | 条件の真偽判定 | 証拠の有無・一致の確認 |
| `read` | 標準入力から値を受け取る | 手順書の途中で相棒に質問する |
| `basename` / `dirname` | パスからファイル名/ディレクトリ部分を取得 | 住所から「建物名」だけ・「町名」だけを読み取る |
| `seq` | 連番を生成する | 証拠品に通し番号を振る |

## 16. コマンドではない学習要素（概念・記号・キー操作）— Phase 2

LPIC では「コマンド名」以上に、シェルの記号や概念の理解が問われる。
これらは特定レベルに縛らず、Mission の中で自然に登場させる。

| 要素 | 実 PC での意味 | ゲーム内の対応 | LPIC |
|---|---|---|---|
| ワイルドカード `*` `?` `[...]` | ファイル名のパターン一致（glob） | **捜索令状の範囲指定**。「`case_` で始まる全ファイルを一括調査」 | 103.3 |
| 引用符 `'` `"` `\` | 空白・特殊文字の保護、変数展開の制御 | スペース入りの偽装ファイル名を開ける唯一の方法 | 103.1 |
| `2>` / `2>&1` | 標準エラー出力の行き先を変える | 「雑音（エラー）」と「声（結果）」は別の管で流れている、を体感 | 103.4 |
| `/dev/null` | 書き込むと消える特殊ファイル | **シュレッダー**。雑音を捨てて声だけ聞く | 103.4 |
| `<`（入力リダイレクト） | ファイルをコマンドの入力にする | 証拠文書を道具に「読み込ませる」 | 103.4 |
| 終了コード（0=成功） | コマンドの成否を数値で返す仕組み | 部下からの「成功(0)/失敗(非0)」報告 | 105.2 |
| 環境変数 `PATH` / `HOME` | コマンド探索経路 / ホームの場所 | 道具箱の場所リスト / 自分の事務所の住所 | 103.1 |
| Tab 補完 | 入力途中の名前を補完 | 相棒の「先読みサポート」（実機でも同じ、と明示） | — |
| `↑` / `↓`（履歴移動） | 過去の入力を呼び出す | 直前の行動をなぞり直す | 103.1 |
| `Ctrl+R` | コマンド履歴の逆検索 | 記憶（履歴）からの想起 | 103.1 |
| `Ctrl+C` | 実行中コマンドの中断 | 緊急停止ボタン（`tail -f` の張り込みの切り上げに必須） | 103.1 |
| `Ctrl+L` | 画面クリア | 机の上を一瞬で片付ける | — |
| FHS（`/etc` `/var` `/home` `/tmp` `/bin`） | Linux 標準のディレクトリ構成 | **街の地図**: 市役所(`/etc`)・公文書館(`/var/log`)・住宅街(`/home`)・ゴミ捨て場(`/tmp`)・道具街(`/bin`) | 104.x |
| 正規表現メタ文字 | パターンを記述する言語 | 犯人の「手口のパターン」を書き表す言葉 | 103.7 |
| `man` のセクション番号 | 1=コマンド、5=ファイル形式 など | 捜査ハンドブックの「章立て」 | — |

**ゲーム設計のポイント:**
Tab 補完・`Ctrl+R`・`↑` はフロントエンド（ターミナル UI `TerminalView.vue`）側の実装。導入すると入力ストレスが激減し、
「実機のターミナルにも同じ機能がある」という発見自体が黒い画面アレルギー克服の成功体験になる。

## 19. 設計時のルール

- ゲーム内の操作結果は、実 PC での操作結果と**意味が同じになる**ようにする
- 「ゲームでしか通用しない操作」は作らない
- コマンドの出力フォーマットは実 Linux に寄せる（完全一致は不要だが、構造は合わせる）
- LPIC の暗記項目（オプション名など）より、「なぜこのコマンドを使うのか」の体験を優先する
