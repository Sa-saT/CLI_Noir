# 🕹️ 探偵風ターミナルゲーム 設計指示書（正式版）

## 📘 プロジェクト概要
本プロジェクトは、Linux コマンド学習用の探偵風 CUI ゲームである。  
プレイヤーは GUI とターミナルを併用し、Mission を進めながら操作スキルを習得する。

- 対象: MVP は Mission1〜3
- UI 表示言語: 日本語統一
- MOC: `moc/index.html` は UI 参考用（本番実装とは分離）

---

## 🧩 技術スタック

### フロントエンド
- Vue 3
- Nuxt（file-based routing を使用）
- Tailwind CSS
- xterm.js（ターミナル表示）

### バックエンド
- Django
- Django REST Framework（HTTP API）
- Django Channels（WebSocket）
- Python（擬似シェル evaluator / 正規表現評価）

### 認証
- JWT（PyJWT）
- HTTP: `Authorization: Bearer <token>`
- WebSocket: query parameter で JWT を付与

---

## 🖼️ UI / UX

画面は 3 領域で構成する。

1. 上部: 現在地、ミッション情報、背景画像
2. 右側: 使用可能コマンド一覧と説明パネル
3. 下部: ターミナル（xterm.js）

追加要件:
- コマンド一覧はレスポンシブで 2〜3 列
- コマンド選択時、説明パネルはフェード表示
- `ssh` / `exit` 時に背景をフェード遷移

## 🧠 仮想ファイルシステム（Virtual FS）

### 保存方針
- ユーザー単位で独立
- `user_id + mission_id` 単位で 1 レコード保存
- DB の JSON カラムで永続化

### JSON スキーマ（MVP）
```json
{
  "current_path": "/root/desk",
  "filesystem": {},
  "remote_mode": false,
  "ssh_host": null,
  "git_state": {
    "staged": [],
    "committed": false,
    "pushed": false
  },
  "mission_flags": {
    "case_checked": false,
    "completed": false
  }
}
```

### 再ログイン時の復元範囲
復元する:
- カレントディレクトリ
- 仮想ファイル構造
- 作成/編集済ファイル
- `remote_mode`, `ssh_host`
- `git_state`
- ミッション進行フラグ

復元しない:
- コマンド入力途中の文字列
- UI スクロール位置
- 一時エラー表示
- 通信中フラグ

## 🔄 local / remote（ssh）挙動

### local 側
例:
```text
root/
├ desk/
├ door/
└ park/
  ├ swing/
  ├ slide/
  ├ chan-up_bar/
  ├ seesaw/
  └ jungle_gym/
```

### remote 側
- `ssh <host>` で疑似接続
- `remote_mode = true`
- local へ戻るには `exit` のみ許可（`cd` では戻れない）

### 接続エラー
- `Permission denied`
- `Host not found`

### 接続先メモ
- 追加確認事項にホスト名の表記ゆれがあるため、実装前に最終一覧を 1 回確定する
  - 候補1: `amusement_park`, `corp_server`, `archive_node`
  - 候補2: `amusement_park`, `school_facility`, `museum_env`

## ✨ WebSocket 仕様（xterm 入力）

### 接続
- 例: `/ws/terminal?token=<JWT>&mission_id=<id>`

### 処理フロー
1. xterm.js からコマンド入力
2. Channels Consumer が受信
3. JWT 検証
4. allowlist 判定
5. evaluator でコマンド実行
6. 仮想 FS / mission state 更新
7. 出力テキストを返却

## 🔐 コマンド実行制御

本プロジェクトは allowlist 方式を採用する。  
詳細は `allowlist_denylist.md` を正とする。

### フェーズ1で実装対象
- 基本: `ls`, `cd`, `pwd`, `touch`, `mkdir`, `cat`, `less`, `echo`, `history`, `clear`
- 検索/整形: `grep`, `find`, `sort`, `uniq`, `awk`
- 接続: `ssh`, `exit`
- 疑似 Git: `git status`, `git add`, `git commit`, `git push`
- 補助: `chmod`, `chown`, `curl`（外部ネットアクセスは不可）

### 禁止コマンド
denylist のコマンドは常に拒否し、エラーを返す。

## 🧮 Mission 判定仕様

- 判定方式: 正規表現ベース
- 順序: 順不同許容
- 文字: 大小文字区別あり
- 不正コマンド時: エラーメッセージ表示のみ（即失敗にはしない）

### 判定データ（例）
```yaml
mission_id: 2
expected_script_patterns:
  - "cat\\s+.+"
  - "grep\\s+-E\\s+.+"
match_policy:
  order: any
  case_sensitive: true
```

---

## 🗃️ 疑似 Git / セーブ仕様

### 進行ルール
- `git add` → `git commit -m "<msg>"` → `git push` の順序必須
- commit message は 1 文字以上（パターン制約なし）
- `git push` 成功時に次ステージ遷移
- 実 Git 連携は行わない（内部状態 `git_state` で判定）

### `git status` 表示
内部状態に応じて以下を返す:
- `Changes staged`
- `No commits yet`
- `Ready to push`
- `Nothing to commit`

## ✅ 受け入れ基準

### 必須
- Mission1〜3 の完了条件が満たせる
- 正常系/異常系で主要コマンドの応答が仕様どおり
- remote 切替・復帰が仕様どおり（`cd` で local へ戻れない）
- 疑似 Git の順序制御が機能する
- 再ログイン時に mission state が復元される

### テスト観点
- 基本ターミナル操作（`ls`, `cd`, `pwd`, `history`, `clear`）
- ファイル操作（`touch`, `mkdir`, `cat`, `less`, `echo`）
- 検索/正規表現（`grep`, `find`, `sort`, `uniq`, `awk`）
- SSH/remote（接続成功、`Host not found`, `Permission denied`, `exit`）
- 疑似 Git（`status/add/commit/push`）
- allowlist/denylist（許可外コマンド拒否）
