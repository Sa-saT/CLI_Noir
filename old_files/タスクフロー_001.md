# CLI_Noir タスクフロー

このファイルは、CLI_Noir を開発する際の標準作業フローを定義する。  
設計・実装・検証・ドキュメント更新を一貫して進めるために使用する。

---

## 1. 事前確認

1. `設計指示書.md` を確認する
2. `Mission参照ファイル.md` の `【🟨 要入力】` が未記入か確認する
3. `allowlist_denylist.md` の最新内容を確認する
4. `バックエンド_コマンド機能仕様.md` の実装項目を確認する
5. `環境構築手順.md` に沿って開発環境が起動できるか確認する

### 1-1. 追加でやっておくと良い事前準備

- `.env.example` を作成し、必要な環境変数を明文化する
- API と WebSocket の疎通確認用に最小エンドポイントを先に用意する
- `missions/` ディレクトリを作成し、`mission1.md` の雛形を先に置く
- エラーメッセージ一覧（例: `Host not found`, `Permission denied`）を先に固定する
- 作業ブランチ命名ルール（例: `feat/mission1-evaluator`）を決める

---

## 2. Mission仕様作成フロー

1. 対象 Mission（1〜5）を決める
2. `Mission参照ファイル.md` の該当セクションを埋める
3. `missions/mission{n}.md` を作成する
4. 以下を必ず記載する
   - 目的
   - 初期状態（current_path / filesystem / remote_mode）
   - 必須コマンド
   - `expected_script_patterns`
   - クリア条件
   - 失敗条件/エラー文言
   - 正常系/異常系テスト

---

## 3. 実装フロー（Frontend / Backend）

### Frontend
1. 画面ルートを作成（`/missions`, `/missions/{id}`）
2. UI 3領域（上部情報 / 右パネル / 下部ターミナル）を実装
3. WebSocket 接続処理（JWT query parameter）を実装
4. コマンド出力と状態表示を連携

### Backend
1. Mission API を実装
   - `GET /api/missions/`
   - `GET /api/missions/{id}/`
   - `POST /api/missions/{id}/complete/`
2. state API を実装（推奨。未実装の場合は理由を記録）
   - `GET /api/missions/{id}/state/`
   - `PUT /api/missions/{id}/state/`
3. evaluator を実装（`バックエンド_コマンド機能仕様.md` を正とする）
   - allowlist/denylist 判定
   - 仮想FS更新
   - 疑似Git状態更新
   - 正規表現判定
4. コマンド実装ごとに `バックエンド_コマンド機能仕様.md` の完了チェックを更新

---

## 4. テストフロー

1. Mission単位で正常系3件、異常系3件を作成
2. コマンドカテゴリごとにテスト
   - 基本操作: `ls`, `cd`, `pwd`, `history`, `clear`
   - ファイル操作: `touch`, `mkdir`, `cat`, `less`, `echo`
   - 検索系: `grep`, `find`, `sort`, `uniq`, `awk`
   - SSH/remote: `ssh`, `exit`
   - 疑似Git: `status`, `add`, `commit`, `push`
3. 不正コマンドで `Error: command not allowed` が返ることを確認
4. 再ログイン時の state 復元を確認

---

## 5. ドキュメント更新フロー

1. 仕様変更がある場合は先に `old_files/` へ退避する
2. 変更対象を更新する
   - `設計指示書.md`
   - `Mission参照ファイル.md`
   - `allowlist_denylist.md`
   - `バックエンド_コマンド機能仕様.md`
   - `環境構築手順.md`
3. Mission仕様変更時は `設計指示書.md` の Mission要約を同期する

---

## 6. 完了条件（Definition of Done）

### 6-1. チェックリスト運用

完了項目は、コード記載・実装進行に合わせてチェックを付ける。

- [ ] Mission仕様が `missions/mission{n}.md` に記載済み
- [ ] API / WebSocket / evaluator が仕様どおり動作
- [ ] 主要コマンドの正常系/異常系テストが通過
- [ ] 再ログイン時の復元が確認済み
- [ ] 関連ドキュメントが最新化されている

---

## 7. 運用メモ

- 迷ったら `設計指示書.md` を正とする
- 実装前に `Mission参照ファイル.md` の `【🟨 要入力】` を先に埋める
- 大きな仕様変更は小分けで反映し、都度ドキュメント同期する
