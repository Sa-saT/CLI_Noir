# 探偵風ターミナルゲーム 設計指示書

## 1. プロジェクト概要

Linux コマンド学習用の探偵風 CUI ゲーム。  
プレイヤーは GUI とターミナルを併用し、Mission を進めながら操作スキルを習得する。

- MVP: Mission1〜3（Mission は段階的に増やしていく前提）
- Phase2: Mission4〜20 採用済み（§ 11。詳細は `Mission参照ファイル.md`）
- UI 表示言語: 日本語統一
- MOC: `moc/index.html` は UI 参考用（本番実装とは分離）
- LPIC 学習との対応は `LPIC学習マップ.md` を参照

### ゲームの趣旨

PC は本来、興味がある人が触りながら徐々に理解し、既存のシステムを変更して理解を深め、  
やがて開発したいと思いスペシャリストになっていったもの。  
このゲームはその「触りながら理解する」原体験を、安全な仮想環境で再現する。

- CLI の良さを再確認する体験を提供する
- 「黒い画面アレルギー」を楽しみながら克服させる
- PC の理論性（なぜそう動くのか）や概念フロー（入力→処理→出力）を、遊びの中で体に染み込ませる
- LPIC 学習の入り口として、教科書だけでは挫折しがちな人が操作感覚を先に掴めるようにする
- 「理解してから触る」ではなく「触りながら理解する」導線にする
- ストーリーと操作の両面で「実際の PC 操作と同じ思考プロセス」を踏ませ、ゲームを終えた時に実機でも同じことができる状態を目指す

### ペルソナ

#### ターゲット: PC の根本に興味がある人（年齢・性別不問）
- 「PC ってどう動いてるんだろう」に好奇心がある
- ターミナルは見たことあるが、自分で使ったことはない / 少しだけ触った
- LPIC の教科書を開いたが、文字だけでは操作イメージが湧かず止まった
- 本やスクールより、まず自分で触って確かめたいタイプ
- 求めていること:
  - 壊れない安全な環境で試行錯誤できること
  - 「何のためにこのコマンドを使うのか」がストーリーで分かること
  - 操作した結果が実 PC と同じ意味を持つこと（ゲーム限定の知識にならない）
  - 小さな成功体験を積み重ねて「自分でもできる」と思えること

#### 副次ターゲット: LPIC 受験予定者
- LPIC Level 1（101/102）の出題範囲に操作で慣れたい
- 教科書の暗記ではなく、操作経験で定着させたい
- コマンドの「意味」と「使い所」をセットで覚えたい

## 2. 技術スタック

### フロントエンド
- Vue 3 / Nuxt（file-based routing）。**SPA モード（`ssr: false`）** — 認証必須 + WebSocket 主体のため SSR の利点がなく、複雑さだけが増える
- Tailwind CSS
- ターミナル UI: **自作ラインベースコンポーネント（`TerminalView.vue`）**。xterm.js は不採用（2026-07-06 改訂）
  - 本ゲームは行単位の入出力のみで全画面 TUI を持たない。xterm.js でもライン編集（エコー・履歴・Tab補完）は結局自作になる
  - 出力は WS から構造化 JSON（style 付き行）で届くため、ANSI エスケープへ変換して xterm に流すより DOM 直描画が単純で、演出（行フェード・強調色）の自由度も高い
  - commit メッセージ等で日本語入力が発生する（例: `git commit -m "探索メモ"`）。native `<input>` は IME 対応が無条件で安全
  - Phase3 で vi を導入する場合に xterm.js を再評価する

### バックエンド
- Django / Django REST Framework（HTTP API）
- Django Channels（WebSocket。dev サーバーは `channels[daphne]` が必要）
- Python（擬似シェル evaluator / 正規表現評価）。**実 subprocess・実ファイルシステムには一切触れない**（仮想FS 上の純粋関数として実装）

### 認証
- JWT（`djangorestframework-simplejwt`。access + refresh。PyJWT を直接組まない）
- HTTP: `Authorization: Bearer <token>`
- WebSocket: **接続後の初回メッセージで認証**（2026-07-06 改訂）
  - `{"type": "auth", "token": "<JWT>"}` を接続後 5 秒以内に送信。届かなければサーバーが切断
  - query parameter 方式は URL がアクセスログ・プロキシに残りトークンが漏洩するため不採用

## 4. 仮想ファイルシステム（Virtual FS）

### 保存方針
- ユーザー単位で独立
- `user_id + mission_id` 単位で 1 レコード
- DB の JSON カラムで永続化

### JSON スキーマ
```json
{
  "current_path": "/root/desk",
  "filesystem": {},
  "remote_mode": false,
  "ssh_host": null,
  "git_state": {
    "staged": [],
    "commits": [
      {
        "id": 1,
        "message": "探索メモ",
        "snapshot": { "current_path": "/root/desk", "filesystem": {}, "mission_flags": {} },
        "created_at": "2026-01-01T12:00:00Z"
      }
    ],
    "pushed": false
  },
  "mission_flags": {
    "case_checked": false,
    "completed": false
  }
}
```

### 仮想FSノードのスキーマ（2026-07-06 追加）

`filesystem` の各ノードは以下の形式を最初から持たせる（Phase2 の `chmod` / `chown` / `ls -l` / `ln` / `su` 導入時にスキーママイグレーションを不要にするため。MVP ではデフォルト値のまま使う）。

```json
{
  "root": {
    "type": "dir",
    "children": {
      "desk": {
        "type": "dir",
        "children": {
          "businesscard.txt": {
            "type": "file",
            "content": "...",
            "mode": "rw-r--r--",
            "owner": "detective",
            "mtime": "2026-01-01T12:00:00Z",
            "immutable": true
          }
        }
      }
    }
  }
}
```

- `type`: `dir` / `file` / `link`（`link` は追加で `target` にリンク先絶対パスを持つ）
- `immutable: true` はデフォルト配置ファイル（§ 8 の「削除不可」ルールと対応）
- パス解決（絶対/相対/`..`）は evaluator 内の単一ヘルパーに集約する

### サイズガード（2026-07-06 追加）

- 1 state レコードの上限: 256KB（超過時は書き込みコマンドに `Error: invalid input` を返す）
- `git_state.commits` の上限: 30 件 / Mission（超過時は最古のクリア前セーブから削除）
- 生成系ファイル（数万行のログ等）は乱数 seed を Mission 定義に固定し、state には seed のみ保存して実体は評価時に生成する（レコード肥大の防止・再現性の確保）

### 復元範囲
復元する: カレントディレクトリ / 仮想FS / 編集済ファイル / remote_mode / ssh_host / git_state（commit履歴含む） / mission_flags  
復元しない: 入力途中文字列 / UIスクロール位置 / 一時エラー / 通信中フラグ

### セーブ選択
再ログイン時、commits 配列から任意のセーブを選んで再開できる。  
次 Mission に遷移した時点で、前 Mission の途中セーブ（クリア前 commit）は全消去される。

## 6. API 仕様

### 大原則（2026-07-06 改訂）

**state とクリア判定の書き込み権限はサーバー側 evaluator（WebSocket）のみが持つ。**
クライアントが state やクリア結果を直接送信できる API は設けない（改ざん・チート防止）。
HTTP API は「読み取り」と「認証」に限定する。

### エンドポイント一覧

| メソッド | パス | 概要 |
|---|---|---|
| POST | `/api/auth/login/` | ログイン（access + refresh 発行） |
| POST | `/api/auth/refresh/` | access トークン再発行 |
| GET | `/api/auth/me/` | ユーザー情報 |
| GET | `/api/missions/` | Mission一覧 |
| GET | `/api/missions/{id}/` | Mission詳細 |
| GET | `/api/missions/{id}/state/` | state取得（セーブ選択 UI 用に commits 一覧含む） |

- ~~`PUT /api/missions/{id}/state/`~~ **廃止**（2026-07-06）: クライアントが `mission_flags` や `current_path` を自由に書けてしまい、クリア条件の改ざんが可能だったため。state 更新は全て WS の evaluator 経由で行う
- ~~`POST /api/missions/{id}/complete/`~~ **廃止**（2026-07-06）: リクエストの `{"passed": true}` をクライアントが自己申告する設計はチート穴。クリア判定・完了記録・次 Mission 解放は `git push` 成功時にサーバー内部で行い、結果は WS の `event` フレーム（§ 7）と `GET /api/missions/` の status 更新で伝える
- セーブ選択（再開 commit の指定）は WS 接続後の `{"type": "resume", "commit_id": n}` フレームで行う（§ 7）

### 詳細

#### `POST /api/auth/login/`
```json
// request
{ "username": "detective01", "password": "secret" }
// response 200
{ "access_token": "<JWT>", "token_type": "Bearer" }
// response 401: 認証失敗
```

#### `GET /api/auth/me/`
```json
// response 200
{ "id": 1, "username": "detective01" }
// response 401: トークン不正/期限切れ
```

#### `GET /api/missions/`
```json
// response 200
[
  { "id": 1, "title": "Edit Business Card", "status": "open" },
  { "id": 2, "title": "Park Cat Search", "status": "locked" },
  { "id": 3, "title": "Amusement Park Bomb", "status": "locked" }
]
```

#### `GET /api/missions/{id}/`
```json
// response 200
{
  "id": 1,
  "title": "Edit Business Card",
  "description": "机上の名刺ファイルを編集せよ",
  "allowed_commands": ["ls", "cd", "cat", "echo", "git"],
  "status": "open"
}
// response 404: ミッション未定義
```

#### `GET /api/missions/{id}/state/`
```json
// response 200
{
  "mission_id": 1,
  "current_path": "/root/desk",
  "remote_mode": false,
  "ssh_host": null,
  "git_state": {
    "staged": [],
    "commits": [ { "id": 1, "message": "探索メモ", "created_at": "..." } ],
    "pushed": false
  },
  "mission_flags": { "case_checked": false, "completed": false }
}
// commits は一覧表示用メタのみ返す（snapshot 本体は返さない）
// response 404: state 未作成（初回はWS接続時にサーバーが初期stateを生成する）
```

## 7. WebSocket 仕様（2026-07-06 改訂）

### 接続とハンドシェイク
`/ws/terminal?mission_id=<id>`（トークンは URL に載せない。§ 2 認証参照）

```json
// 1. クライアント → サーバー（接続後 5 秒以内。なければ切断）
{ "type": "auth", "token": "<JWT>" }

// 2. サーバー → クライアント（認証成功。現在 state とセーブ一覧を返す）
{ "type": "hello",
  "state": { "current_path": "/root", "remote_mode": false, "ssh_host": null },
  "commits": [ { "id": 1, "message": "探索メモ", "created_at": "..." } ] }

// 3.（任意）セーブ選択。省略時は最新 state で再開
{ "type": "resume", "commit_id": 1 }
```

### コマンド実行フレーム
```json
// クライアント → サーバー（id はクライアント採番。応答の突き合わせ用）
{ "type": "exec", "id": 42, "command": "ls /root" }

// サーバー → クライアント（正常）
{ "type": "result", "id": 42, "ok": true, "command": "ls /root",
  "lines": [
    { "text": "desk", "style": "normal" },
    { "text": "door", "style": "normal" },
    { "text": "park", "style": "normal" }
  ],
  "state": { "current_path": "/root", "remote_mode": false, "ssh_host": null } }

// サーバー → クライアント（エラー）
{ "type": "result", "id": 43, "ok": false, "command": "wget http://x",
  "lines": [ { "text": "Error: command not allowed", "style": "error" } ] }
```

- `style`: `normal` / `error` / `warning` / `emphasis` / `success`。`docs/DESIGN.md` § 2 の色トークンに 1:1 対応（フロントは style→CSS クラスの変換だけを行い、色の意味付けはサーバーが決める）

### サーバープッシュフレーム（Phase2）
```json
// tail -f のライブ監視・ランクアップ通知など
{ "type": "stream", "source": "tape.log", "lines": [ ... ] }
{ "type": "event", "name": "mission_clear", "next_mission_id": 2 }
{ "type": "event", "name": "rank_up", "level": 5, "unlocked": ["head", "tail"] }
```

### 処理フロー
1. ターミナル UI（`TerminalView.vue`）からコマンド行を `exec` フレームで送信
2. Channels Consumer が受信（未認証の接続は即切断）
3. denylist → allowlist 判定
4. evaluator でコマンド実行（仮想FS 上の純粋関数。実 OS には触れない）
5. 仮想 FS / mission state を更新し DB へ保存
6. `result` フレームを返却（クリア時は `event: mission_clear` も送出）

### 実装ノート
- MVP は「1 接続 = 1 Consumer の直接応答」で完結するため **channel layer（Redis）は不要**。`InMemoryChannelLayer` すら使わず Consumer が直接 send してよい。Redis はマルチプロセス構成 + サーバープッシュの本格導入時に検討
- 切断・再接続時は `auth` → `hello` からやり直す（`hello` の state が常に正。フロント側でのステート推測は禁止）

## 8. コマンド実行制御（allowlist / denylist）

本プロジェクトは allowlist 方式を採用する。  
コマンドごとの実装詳細は `バックエンド_コマンド機能仕様.md` を正とする。

### 判定順序
`denylist 判定 → allowlist 判定 → コマンド実行`

`git` は第1トークンで判定後、サブコマンド（`status/add/commit/push`）を別途判定する。

### allowlist（決定版）
```python
ALLOWLIST = [
    # Level 1〜4（MVP）
    "ls", "cd", "pwd",
    "touch", "mkdir", "cat", "less",
    "echo", "history", "clear",
    "grep", "find", "sort", "uniq", "awk",
    "ssh", "exit",
    "git",
    "chmod", "chown", "curl",
    # Level 5: テキスト処理・パイプ（Phase2）
    "head", "tail", "wc", "cut", "paste", "tr", "sed", "diff", "nl", "tee", "xargs",
    # Level 6: プロセス（Phase2）※仮想プロセステーブル限定
    "ps", "top", "kill", "pgrep", "jobs", "free", "uptime",
    # Level 7: 権限・ユーザー（Phase2）
    "umask", "su", "whoami", "id", "who",
    # Level 8: ファイル管理・アーカイブ・鑑識（Phase2）
    "cp", "mv",
    "tar", "gzip", "gunzip", "zip", "unzip", "ln", "file",
    "which", "locate", "du", "df", "md5sum", "sha256sum", "stat",
    # Level 9: ネットワーク（Phase2）
    "ping", "ip", "ss", "dig", "host", "hostname", "traceroute",
    # Level 10: システム管理・自動化（Phase2）
    "crontab", "at", "date", "cal", "systemctl", "journalctl", "uname", "env", "alias",
    # Level 11: シェルスクリプト基礎（Phase2）
    "sh", "test", "read", "basename", "dirname", "seq",
    # 全レベル共通（Phase2）
    "man", "apropos", "whatis",
]
```

### 構文レベルの許可（Phase2・evaluator パーサー対応）
コマンド名の判定とは別に、以下のシェル構文を evaluator で解釈する。
- glob（`*` `?` `[...]`）/ 引用符（`'` `"` `\`）
- リダイレクト `2>` `2>&1` `<`（`>` `>>` は既存）と `/dev/null`
- 終了コード `$?` / 変数定義・展開 / `if` `for` `test`（サブセット実装）

### denylist（決定版）
```python
DENYLIST = [
    "rm", "sudo", "shutdown",
    "poweroff", "reboot",
    "telnet", "ftp", "wget",
    "nc", "dd", "mkfs",
]
```

### レベル分け（ミッション段階開放・探偵ランク制度）
レベル＝探偵ランクとして演出し、Mission クリアでランクアップ → 新コマンド解放とする。

| レベル | ランク名 | コマンド | Phase |
|---|---|---|---|
| Level 1 | 見習い探偵 | `ls`, `cd`, `pwd`, `touch`, `mkdir`, `cat`, `echo`, `clear` | 1 |
| Level 2 | 新米探偵 | `less`, `history` | 1 |
| Level 3 | 捜査員 | `grep`, `find`, `sort`, `uniq` | 1 |
| Level 4 | 分析官 | `awk` | 1 |
| Level 5 | 情報屋 | パイプ, `head`, `tail`, `wc`, `cut`, `paste`, `tr`, `sed`, `diff`, `nl`, `tee`, `xargs` | 2 |
| Level 6 | 監視者 | `ps`, `top`, `kill`, `pgrep`, `jobs`, `free`, `uptime` | 2 |
| Level 7 | 潜入捜査官 | `chmod`, `chown`, `umask`, `su`, `whoami`, `id`, `who` | 2 |
| Level 8 | 証拠管理官 | `cp`, `mv`, `tar`, `gzip`, `gunzip`, `zip`, `unzip`, `ln`, `file`, `which`, `locate`, `du`, `df`, `md5sum`, `sha256sum`, `stat` | 2 |
| Level 9 | 追跡者 | `ping`, `ip`, `ss`, `dig`, `host`, `hostname`, `traceroute` | 2 |
| Level 10 | 主任探偵 | `crontab`, `at`, `date`, `cal`, `systemctl`, `journalctl`, `uname`, `env`, `alias` | 2 |
| Level 11 | 参謀 | `sh`, `test`, `read`, `basename`, `dirname`, `seq`, 変数, `if`/`for` | 2 |
| 全レベル | — | `ssh`, `exit`, `man`, `apropos`, `whatis` | 1（man 系は 2） |
| Git 系 | — | `git status/add/commit/push` | 1 |

### 補足ルール
- `curl` は mock API 呼び出し用に限定（外部ネットアクセス不可）
- `touch` で作成したファイルは UI 削除ボタンで削除可能
- デフォルト配置ファイルは削除不可
- `rm` は全般禁止
- `kill` は仮想プロセステーブル内のプロセス限定（実プロセスには一切作用しない）
- `sudo` は denylist 維持。導入する場合も最終 Mission のイベント演出限定とし、自由実行は許可しない

## 9. Mission 判定仕様

- 判定方式: 正規表現ベース
- 順序: 順不同許容
- 文字: 大小文字区別あり
- 不正コマンド時: エラーメッセージ表示のみ（即失敗にしない）

### 判定データ例
```yaml
mission_id: 2
expected_script_patterns:
  - "cat\\s+.+"
  - "grep\\s+-E\\s+.+"
match_policy:
  order: any
  case_sensitive: true
```

## 10. 疑似 Git / セーブ仕様

### 基本ルール
- `git commit` ＝ **ゲームセーブ**（1 Mission 中に何度でも可）
- `git push` ＝ **クリア判定**（最新セーブの状態で合否を判定）
- 実 Git 連携は行わない（`git_state` で判定）

### 判定フロー
```
① sh case_file.sh   → 判定実行。OK なら case_checked = true
② git add            → case_file の OK 状態を staged に上げる
③ git commit -m "…"  → ②までの状態をセーブ（snapshot として保存。何度でも可）
④ git push           → 最新 commit の snapshot を判定
                         → case_checked=true なら Next Mission 解放
                         → 条件不足なら Error
```

### セーブ仕様
- 各 commit は状態の snapshot（current_path / filesystem / mission_flags）を保持する
- 再ログイン時、commits 一覧から任意のセーブを選んで再開できる
- 次 Mission に遷移すると、前 Mission のクリア前 commit（途中セーブ）は全消去される

### `git status` 表示
| 内部状態 | 表示 |
|---|---|
| 変更なし（add 前） | `Nothing to commit` |
| add 済み | `Changes staged` |
| commit 未（add はある） | `No commits yet` |
| commit 済み + case_checked | `Ready to push` |

## 12. エラーメッセージ一覧

### 共通
| メッセージ | 用途 |
|---|---|
| `Error: command not allowed` | denylist/allowlist外 |
| `Error: invalid input` | 引数不正 |
| `Error: path not found` | パス不正 |
| `Error: file not found` | ファイル未存在 |
| `Error: directory not found` | ディレクトリ未存在 |
| `Error: directory already exists` | mkdir 重複 |
| `Error: invalid pattern` | 正規表現不正 |
| `Error: pattern mismatch` | 判定致命不一致 |
| `Warning: pattern mismatch` | 判定軽度不一致 |

### SSH / remote
| メッセージ | 用途 |
|---|---|
| `Host not found` | 未登録ホスト |
| `Permission denied` | 権限不可 |
| `Error: remote not connected` | remote未接続でremoteファイル参照 |

### 疑似 Git
| メッセージ | 用途 |
|---|---|
| `Error: nothing to commit` | add なしで commit |
| `Error: commit message required` | message なし |

|---|---|
