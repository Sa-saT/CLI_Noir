# 探偵風ターミナルゲーム 設計指示書

## 1. プロジェクト概要

Linux コマンド学習用の探偵風 CUI ゲーム。  
プレイヤーは GUI とターミナルを併用し、Mission を進めながら操作スキルを習得する。

- MVP: Mission1〜3（Mission は段階的に増やしていく前提）
- UI 表示言語: 日本語統一
- MOC: `moc/index.html` は UI 参考用（本番実装とは分離）
- LPIC 学習との対応は `LPIC学習マップ.md` を参照

### ゲームの趣旨

PC は本来、興味がある人が触りながら徐々に理解し、既存のシステムを変更して理解を深め、  
やがて開発したいと思いスペシャリストになっていったもの。  
このゲームはその「触りながら理解する」原体験を、安全な仮想環境で再現する。

- CLI の良さを再確認する体験を提供する
- 「黒い画面アレルギー」を楽しみながら克服させる
- PC の理論性（なぜそう動くのか）や概念フロー（入力→処理→出力）を、遊びの中で体に染み込ませる
- LPIC 学習の入り口として、教科書だけでは挫折しがちな人が操作感覚を先に掴めるようにする
- 「理解してから触る」ではなく「触りながら理解する」導線にする
- ストーリーと操作の両面で「実際の PC 操作と同じ思考プロセス」を踏ませ、ゲームを終えた時に実機でも同じことができる状態を目指す

### ペルソナ

#### ターゲット: PC の根本に興味がある人（年齢・性別不問）
- 「PC ってどう動いてるんだろう」に好奇心がある
- ターミナルは見たことあるが、自分で使ったことはない / 少しだけ触った
- LPIC の教科書を開いたが、文字だけでは操作イメージが湧かず止まった
- 本やスクールより、まず自分で触って確かめたいタイプ
- 求めていること:
  - 壊れない安全な環境で試行錯誤できること
  - 「何のためにこのコマンドを使うのか」がストーリーで分かること
  - 操作した結果が実 PC と同じ意味を持つこと（ゲーム限定の知識にならない）
  - 小さな成功体験を積み重ねて「自分でもできる」と思えること

#### 副次ターゲット: LPIC 受験予定者
- LPIC Level 1（101/102）の出題範囲に操作で慣れたい
- 教科書の暗記ではなく、操作経験で定着させたい
- コマンドの「意味」と「使い所」をセットで覚えたい

## 4. 仮想ファイルシステム（Virtual FS）

### 保存方針
- ユーザー単位で独立
- `user_id + mission_id` 単位で 1 レコード
- DB の JSON カラムで永続化

### JSON スキーマ
```json
{
  "current_path": "/root/desk",
  "filesystem": {},
  "remote_mode": false,
  "ssh_host": null,
  "git_state": {
    "staged": [],
    "commits": [
      {
        "id": 1,
        "message": "探索メモ",
        "snapshot": { "current_path": "/root/desk", "filesystem": {}, "mission_flags": {} },
        "created_at": "2026-01-01T12:00:00Z"
      }
    ],
    "pushed": false
  },
  "mission_flags": {
    "case_checked": false,
    "completed": false
  }
}
```

### 復元範囲
復元する: カレントディレクトリ / 仮想FS / 編集済ファイル / remote_mode / ssh_host / git_state（commit履歴含む） / mission_flags  
復元しない: 入力途中文字列 / UIスクロール位置 / 一時エラー / 通信中フラグ

### セーブ選択
再ログイン時、commits 配列から任意のセーブを選んで再開できる。  
次 Mission に遷移した時点で、前 Mission の途中セーブ（クリア前 commit）は全消去される。

## 6. API 仕様

### エンドポイント一覧

| メソッド | パス | 概要 |
|---|---|---|
| POST | `/api/auth/login/` | ログイン |
| GET | `/api/auth/me/` | ユーザー情報 |
| GET | `/api/missions/` | Mission一覧 |
| GET | `/api/missions/{id}/` | Mission詳細 |
| POST | `/api/missions/{id}/complete/` | Mission完了 |
| GET | `/api/missions/{id}/state/` | state取得 |
| PUT | `/api/missions/{id}/state/` | state更新 |

### 詳細

#### `POST /api/auth/login/`
```json
// request
{ "username": "detective01", "password": "secret" }
// response 200
{ "access_token": "<JWT>", "token_type": "Bearer" }
// response 401: 認証失敗
```

#### `GET /api/auth/me/`
```json
// response 200
{ "id": 1, "username": "detective01" }
// response 401: トークン不正/期限切れ
```

#### `GET /api/missions/`
```json
// response 200
[
  { "id": 1, "title": "Edit Business Card", "status": "open" },
  { "id": 2, "title": "Park Cat Search", "status": "locked" },
  { "id": 3, "title": "Amusement Park Bomb", "status": "locked" }
]
```

#### `GET /api/missions/{id}/`
```json
// response 200
{
  "id": 1,
  "title": "Edit Business Card",
  "description": "机上の名刺ファイルを編集せよ",
  "allowed_commands": ["ls", "cd", "cat", "echo", "git"],
  "status": "open"
}
// response 404: ミッション未定義
```

#### `POST /api/missions/{id}/complete/`
```json
// request
{ "passed": true, "summary": "Mission Complete!" }
// response 200
{ "mission_id": 1, "completed": true, "next_mission_id": 2 }
// response 409: 完了条件未達
```

#### `GET /api/missions/{id}/state/`
```json
// response 200
{
  "mission_id": 1,
  "current_path": "/root/desk",
  "remote_mode": false,
  "ssh_host": null,
  "git_state": { "staged": [], "committed": false, "pushed": false },
  "mission_flags": { "case_checked": false, "completed": false }
}
```

#### `PUT /api/missions/{id}/state/`
```json
// request
{ "current_path": "/root/park", "remote_mode": false, "ssh_host": null }
// response 200: 更新後 state
// response 400: 入力形式エラー
```

## 7. WebSocket 仕様

### 接続
`/ws/terminal?token=<JWT>&mission_id=<id>`

### 処理フロー
1. xterm.js からコマンド入力
2. Channels Consumer が受信
3. JWT 検証
4. denylist → allowlist 判定
5. evaluator でコマンド実行
6. 仮想 FS / mission state 更新
7. 結果を返却

### 返却フォーマット
```json
// 正常
{ "ok": true, "command": "ls /root", "output": ["desk", "door", "park"],
  "state": { "current_path": "/root", "remote_mode": false, "ssh_host": null } }
// エラー
{ "ok": false, "command": "wget http://x", "error": "Error: command not allowed" }
```

## 8. コマンド実行制御（allowlist / denylist）

本プロジェクトは allowlist 方式を採用する。  
コマンドごとの実装詳細は `バックエンド_コマンド機能仕様.md` を正とする。

### 判定順序
`denylist 判定 → allowlist 判定 → コマンド実行`

`git` は第1トークンで判定後、サブコマンド（`status/add/commit/push`）を別途判定する。

### allowlist（決定版）
```python
ALLOWLIST = [
    "ls", "cd", "pwd",
    "touch", "mkdir", "cat", "less",
    "echo", "history", "clear",
    "grep", "find", "sort", "uniq", "awk",
    "ssh", "exit",
    "git",
    "chmod", "chown", "curl",
]
```

### denylist（決定版）
```python
DENYLIST = [
    "rm", "sudo", "shutdown",
    "poweroff", "reboot",
    "telnet", "ftp", "wget",
    "nc", "dd", "mkfs",
]
```

### レベル分け（ミッション段階開放）
| レベル | コマンド |
|---|---|
| Level 1 | `ls`, `cd`, `pwd`, `touch`, `mkdir`, `cat`, `echo`, `clear` |
| Level 2 | `less`, `history` |
| Level 3 | `grep`, `find`, `sort`, `uniq` |
| Level 4 | `awk` |
| 全レベル | `ssh`, `exit` |
| Git 系 | `git status/add/commit/push` |

### 補足ルール
- `curl` は mock API 呼び出し用に限定（外部ネットアクセス不可）
- `touch` で作成したファイルは UI 削除ボタンで削除可能
- デフォルト配置ファイルは削除不可
- `rm` は全般禁止

## 9. Mission 判定仕様

- 判定方式: 正規表現ベース
- 順序: 順不同許容
- 文字: 大小文字区別あり
- 不正コマンド時: エラーメッセージ表示のみ（即失敗にしない）

### 判定データ例
```yaml
mission_id: 2
expected_script_patterns:
  - "cat\\s+.+"
  - "grep\\s+-E\\s+.+"
match_policy:
  order: any
  case_sensitive: true
```

## 10. 疑似 Git / セーブ仕様

### 基本ルール
- `git commit` ＝ **ゲームセーブ**（1 Mission 中に何度でも可）
- `git push` ＝ **クリア判定**（最新セーブの状態で合否を判定）
- 実 Git 連携は行わない（`git_state` で判定）

### 判定フロー
```
① sh case_file.sh   → 判定実行。OK なら case_checked = true
② git add            → case_file の OK 状態を staged に上げる
③ git commit -m "…"  → ②までの状態をセーブ（snapshot として保存。何度でも可）
④ git push           → 最新 commit の snapshot を判定
                         → case_checked=true なら Next Mission 解放
                         → 条件不足なら Error
```

### セーブ仕様
- 各 commit は状態の snapshot（current_path / filesystem / mission_flags）を保持する
- 再ログイン時、commits 一覧から任意のセーブを選んで再開できる
- 次 Mission に遷移すると、前 Mission のクリア前 commit（途中セーブ）は全消去される

### `git status` 表示
| 内部状態 | 表示 |
|---|---|
| 変更なし（add 前） | `Nothing to commit` |
| add 済み | `Changes staged` |
| commit 未（add はある） | `No commits yet` |
| commit 済み + case_checked | `Ready to push` |

## 12. エラーメッセージ一覧

### 共通
| メッセージ | 用途 |
|---|---|
| `Error: command not allowed` | denylist/allowlist外 |
| `Error: invalid input` | 引数不正 |
| `Error: path not found` | パス不正 |
| `Error: file not found` | ファイル未存在 |
| `Error: directory not found` | ディレクトリ未存在 |
| `Error: directory already exists` | mkdir 重複 |
| `Error: invalid pattern` | 正規表現不正 |
| `Error: pattern mismatch` | 判定致命不一致 |
| `Warning: pattern mismatch` | 判定軽度不一致 |

### SSH / remote
| メッセージ | 用途 |
|---|---|
| `Host not found` | 未登録ホスト |
| `Permission denied` | 権限不可 |
| `Error: remote not connected` | remote未接続でremoteファイル参照 |

### 疑似 Git
| メッセージ | 用途 |
|---|---|
| `Error: nothing to commit` | add なしで commit |
| `Error: commit message required` | message なし |
| `Error: push not allowed before commit` | commit なしで push |
| `Error: mission requirements not met` | push 時にクリア条件未達 |

### 認証 / API
| メッセージ | 用途 |
|---|---|
| `Error: unauthorized` | 認証失敗 |
| `Error: token expired` | JWT期限切れ |
| `Error: mission not found` | Mission未定義 |

## 13. 受け入れ基準

### 必須
- Mission1〜3 の完了条件が満たせる
- 正常系/異常系で主要コマンドの応答が仕様どおり
- remote 切替・復帰が仕様どおり
- 疑似 Git の順序制御が機能する
- 再ログイン時に mission state が復元される

### テスト観点
- 基本操作: `ls`, `cd`, `pwd`, `history`, `clear`
- ファイル操作: `touch`, `mkdir`, `cat`, `less`, `echo`
- 検索/正規表現: `grep`, `find`, `sort`, `uniq`, `awk`
- SSH/remote: 接続成功、`Host not found`、`Permission denied`、`exit`
- 疑似 Git: `status/add/commit/push`
- allowlist/denylist: 許可外コマンド拒否

## 14. 開発フロー

### 事前確認
1. 本ファイル（`設計指示書.md`）を確認する
2. `Mission参照ファイル.md` の未確定項目を確認する
3. `バックエンド_コマンド機能仕様.md` の実装項目を確認する
4. `環境構築手順.md` に沿って開発環境が起動できるか確認する

### 実装フロー

#### Frontend
1. 画面ルートを作成（`/missions`, `/missions/{id}`）
2. UI 3 領域を実装
3. WebSocket 接続処理を実装
4. コマンド出力と状態表示を連携

#### Backend
1. 認証 API / Mission API を実装
2. state API を実装
3. evaluator を実装（`バックエンド_コマンド機能仕様.md` を正とする）
4. コマンド実装ごとに `バックエンド_コマンド機能仕様.md` の完了チェックを更新

### テストフロー
1. Mission 単位で正常系 3 件、異常系 3 件を作成
2. コマンドカテゴリごとにテスト
3. 不正コマンドで `Error: command not allowed` が返ることを確認
4. 再ログイン時の state 復元を確認

### 完了条件（DoD）
- [ ] API / WebSocket / evaluator が仕様どおり動作
- [ ] 主要コマンドの正常系/異常系テストが通過
- [ ] 再ログイン時の復元が確認済み
- [ ] 関連ドキュメントが最新化されている

### ドキュメント更新ルール
- 仕様変更時は先に `old_files/` へ退避してから更新する
- 迷ったら `設計指示書.md`（本ファイル）を正とする
